# F08-ai-curator Design

## Chosen Approach

F08 ships a deterministic, rule/tag-based museum curator before the later embedding pipeline exists. The curator accepts public artwork metadata and returns wing themes, room assignments, layout slots, accessible text summaries, and visible grouping reasons. It does not use private artist attributes and does not expose sensitive metadata in room labels.

The algorithm is intentionally transparent:

1. Normalize each artwork into public curation metadata: id, title, artist display name, category, materials, tags, impact estimate, color palette, location at city/country granularity, and image URL.
2. Assign each artwork to a primary theme using category first, then material, then impact strength.
3. Place sculpture/installation works in court-style rooms, wall art and mixed media in main galleries, functional/home decor in side galleries, and jewellery/other compact works in cabinet rooms.
4. Sort deterministically by theme priority, impact score descending, title, and id.
5. Allocate slots with stable capacity rules so the same inputs always produce the same rooms and slots.
6. Emit `accessibilitySummary` and `arrangementExplanation` strings for screen-reader and flat-screen fallback use.

## Alternatives Considered

- Embedding clustering with CLIP/text embeddings: deferred to Phase 2 because F02 material vision and tagging are not available yet.
- Random procedural placement: rejected because acceptance requires deterministic layout from the same inputs.
- Direct Three.js-only placement logic: rejected because tests and future API reuse require a pure curation module independent of rendering.

## Data Schema Changes

No database migration is required in this unit. F08 introduces API-facing TypeScript types derived from the existing `ArtworkRecord` schema:

- `CurationArtworkInput`: public artwork metadata used for placement.
- `CuratedMuseumRoom`: room theme, room kind, visible grouping criteria, capacity, and accessibility label.
- `CuratedArtworkPlacement`: artwork id, room id, wing index, slot index, wall side, and explanation.
- `MuseumCurationPlan`: full deterministic output.

Future TODO for F02/P2: add optional image/text embedding ids and cluster labels once the tagging pipeline exists.

## API Surface

- `POST /api/museum/curation`
  - Input: `{ artworks: CurationArtworkInput[] }`
  - Output: `{ plan, requestId }`
  - Validation: rejects malformed JSON, missing arrays, unknown categories/materials, private metadata fields, invalid URLs, unrealistic impact values, and more than 500 artworks per request.
  - Rate limit: P0 in-memory public endpoint limiter.

The virtual museum integration should use the same curation module for room and artwork placements instead of maintaining separate category-only logic.

## Model Card

This is not a trained ML model. It is a transparent rule-based curation assistant.

- Intended use: arrange public artwork listings into coherent museum rooms by type, material, and impact.
- Out-of-scope use: ranking artist quality, judging cultural value, or making moderation decisions.
- Bias risks: deterministic grouping can overemphasize categories with more inventory. The implementation includes an emerging-room strategy by material/impact but does not yet enforce demographic fairness because protected attributes are intentionally not used in this public placement unit.
- Human oversight: curators can override generated placement in later dashboard units.

## Failure Modes

- Sparse metadata can place works in generic rooms.
- Very large single-category inventories create many repeated rooms.
- Missing impact estimates reduce impact-based grouping strength.
- Material spelling mismatches are rejected rather than silently coerced.
- Client-side museum rendering can drift if it bypasses the shared module.

## Rollback Plan

The curation API and module are additive. Rollback removes `/api/museum/curation`, `src/lib/ml/curator.ts`, tests, open artifacts, and the virtual-room import. Existing pricing and impact APIs remain unaffected.

## Acceptance Mapping

- Museum rooms generated by metadata: curation module creates room ids/themes from category/material/impact.
- Type/material/impact arrangement visible: room labels and placement explanations include these criteria.
- Deterministic same inputs: pure function with stable sorting and tests.
- Accessibility: output includes text summaries and room labels; existing virtual-room keyboard/mouse navigation remains available.
